### Motivation
Developers don't work in isolation. You'll be part of teams—code reviewing each other's work, pair programming, onboarding new features, and discussing technical decisions. AI changes the dynamics of collaboration.

Here's the tension: AI is powerful, but it's also **opaque**. When you submit code to a pull request that was partially generated by AI, teammates might wonder:
- *Is this really your solution, or did AI write it?*
- *Can I trust this code if I don't know how it was generated?*
- *Are we becoming dependent on AI instead of thinking deeply?*

These aren't paranoid concerns—they're legitimate questions about code quality, ownership, and team trust.

You'll learn to:
1. **Communicate your AI usage transparently** without over-explaining or apologizing
2. **Integrate AI-assisted code into code review processes** responsibly
3. **Use AI to enhance collaboration** (documentation, problem-solving, knowledge sharing)
4. **Build team guidelines** for responsible AI use
5. **Discuss AI in interviews** as a team player who knows when and how to use tools

Teams value engineers who can collaborate effectively with AI *and* with people. Mastering collaborative AI use makes you indispensable in team settings.

### Objectives
By the end of this session, you will be able to:
1. **Articulate your AI usage** clearly in code reviews, pull request descriptions, and team conversations.
2. **Evaluate team dynamics** around AI use and identify potential friction points before they become problems.
3. **Use AI to improve collaboration** (generating summaries, explaining complex code, facilitating knowledge sharing).
4. **Propose and discuss team guidelines** for responsible AI use in your organization.
5. **Navigate ethical and practical concerns** about AI in code review and team settings.
6. **Frame your experience with collaborative AI** in interviews and team conversations.

### Specific Things To Learn

#### Communication & Transparency
- **When to disclose AI usage:** Code review, pull request descriptions, team meetings, discussions, feature ideation
- **How to disclose without over-explaining:** Clear statements that balance transparency and confidence
- **Framing AI as a tool, not a crutch:** "I used AI to generate a first draft, then refined it" vs. "AI wrote this"
- **Addressing team concerns:** How to respond when teammates question AI-generated code
- **Documentation and audit trails:** Leaving evidence of your reasoning and verification process

#### Code Review with AI
- **Reviewing AI-generated code:** What to look for, what to verify, what to challenge
- **Receiving code review feedback on AI-assisted work:** Understanding criticism and incorporating suggestions
- **Using AI in code review:** Asking AI to explain code, suggest improvements, identify issues
- **Maintaining code quality standards:** Ensuring AI doesn't become an excuse for lower standards
- **Security and performance review:** Red flags specific to AI-generated code

#### Collaboration & Team Dynamics
- **Pair programming with AI:** How AI tools can or cannot support paired work
- **Knowledge sharing:** Using AI to document complex features for teammates
- **Onboarding:** AI-generated documentation to help new team members understand code
- **Problem-solving together:** Collaborative use of AI to tackle complex issues
- **Trust and psychological safety:** Building team culture where AI is a tool, not a threat

#### Team Guidelines & Governance
- **Proposing AI use policies:** When AI is appropriate, when it's not (security-critical code, interview take-homes, etc.)
- **Balancing speed and learning:** Using AI to accelerate routine work without stunting growth
- **Organizational context:** Different guidelines for startups vs. enterprises, regulated industries, etc.
- **Open discussion:** Creating spaces where teammates can voice concerns about AI

#### Activity 1: Simulating Code Review with AI-Assisted Code (18 minutes)

**Objective:** Practice reviewing AI-generated code critically and communicating findings as if you're in a real code review.

**Task:**
You'll be given AI-generated code (simulating code that a teammate submitted in a pull request). Your job is to:
1. Review the code as if you're a peer reviewer
2. Identify strengths and concerns
3. Draft feedback that's constructive and considers the AI aspect
4. Document how you'd communicate this to the teammate

**Choose your language:**

**Option A — JavaScript (React Component with AI-Assisted Generation):**

**PR Description (from teammate):**
> "Used AI to help generate a user profile component. I reviewed the output and made edits. Please review."

<details><summary>Code in PR:</summary>

```javascript
// UserProfile.jsx - Generated with AI assistance, then reviewed
import React, { useState, useEffect } from 'react';

export function UserProfile({ userId }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    fetch(`/api/users/${userId}`)
      .then(res => res.json())
      .then(data => {
        setUser(data);
        setLoading(false);
      })
      .catch(err => console.log('Error:', err)); // AI-generated; not ideal
  }, [userId]);

  if (loading) return <div>Loading...</div>;
  if (!user) return <div>User not found</div>;

  return (
    <div className="profile">
      <h1>{user.name}</h1>
      <p>Email: {user.email}</p>
      <p>Bio: {user.bio || 'No bio provided'}</p>
      <button onClick={() => alert('Edit functionality not yet implemented')}>
        Edit Profile
      </button>
    </div>
  );
}
```

**Option B — Python (FastAPI Endpoint with AI-Assisted Generation):**

**PR Description (from teammate):**
> "Used AI to help generate a user profile endpoint. Reviewed the code carefully. Let me know if changes are needed."

<details><summary>Code in PR:</summary>

```python
# user_profile.py - Generated with AI assistance, then reviewed
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel

app = FastAPI()

class UserProfile(BaseModel):
    id: int
    name: str
    email: str
    bio: str = None

# Mock database
users_db = {
    1: {"id": 1, "name": "Alice", "email": "alice@example.com", "bio": "Engineer"},
    2: {"id": 2, "name": "Bob", "email": "bob@example.com", "bio": None},
}

@app.get("/users/{user_id}", response_model=UserProfile)
async def get_user(user_id: int):
    if user_id not in users_db:
        raise HTTPException(status_code=404, detail="User not found")
    return users_db[user_id]

@app.put("/users/{user_id}")
async def update_user(user_id: int, user: UserProfile):
    if user_id not in users_db:
        raise HTTPException(status_code=404, detail="User not found")
    users_db[user_id] = user.dict()
    return {"message": "User updated"}
```
</details>

**Step 1: Read & Understand the Code (3 minutes)**
Read through the code carefully. Note:
- What does this code do?
- How is it structured?
- Are there any obvious issues?

**Deliverable:** Write a 2-3 sentence summary of what this code does and its general approach.

**Step 2: Identify Strengths (3 minutes)**
What did the AI (or the author) get right? Consider:
- Code structure and organization
- Readability and naming
- Basic functionality
**Deliverable:** List 2-3 strengths. Example format:

```plaintext
Strength 1: Component is well-structured with clear separation of concerns
Strength 2: Uses appropriate React hooks (useState, useEffect)
Strength 3: Handles loading and error states visually
```

**Step 3: Identify Concerns (5 minutes)**
What issues do you notice? Consider:
- **Error handling:** Is it robust?
- **Performance:** Could there be issues (unnecessary re-renders, N+1 queries)?
- **Security:** Are there vulnerabilities?
- **Testing:** How would you test this?
- **Maintainability:** Would this be easy to modify or extend?
- **AI-specific red flags:** Does something look like "plausible but incorrect" output?

**Deliverable:** List 3-4 concerns. Example format:

```plaintext
Concern 1: Error handling uses console.log instead of proper error state
Why it matters: Errors won't be visible to users; debugging will be difficult
Suggested fix: Add error state and display error message

Concern 2: Missing dependency in useEffect ([userId]) - could cause stale closures
Why it matters: If userId changes, effect won't re-run
Suggested fix: Add userId to dependency array (already correct in the code)

Concern 3: No loading state after component mounts; could flash "User not found"
Why it matters: Poor UX; looks like an error when it's actually loading
Suggested fix: Add proper loading/skeleton states
```

**Step 4: Draft Constructive Feedback (5 minutes)**
Now write **feedback as if you're a peer reviewer**. Your tone should be:
- Collaborative (not confrontational)
- Specific (not vague)
- Acknowledging the work (including AI usage)
- Actionable (clear suggestions)

**Deliverable:** Draft a code review comment. Example format:

```plaintext
Overall: Thanks for the PR! The component structure is clear and well-organized. 
I can see you used AI to generate this—nice that you reviewed it carefully. 
I have a few suggestions to strengthen it:

1. Error Handling
   - Current: console.log('Error:', err)
   - Issue: Errors won't be visible to users
   - Suggestion: Add an error state (setError) and display it in the UI
   - Why: Users should know when something went wrong, not silently fail

2. Mock Data Consideration (if using mock DB)
   - Question: Is this using real API or mock data? If mock, should we add a comment?
   - Reason: Future readers should know this is for demo purposes
   
3. Tests
   - Have you written tests for this component? (loading state, error state, success state)
   - I can help review if you want to add them

Would love to discuss any of these. Great work getting this started!
```

**Step 5: Reflect on Your Feedback (2 minutes)**
Write a 2-3 sentence reflection:
- How would you communicate the AI usage to your reviewer?
- Did you feel defensive about the AI assistance, or confident in your review?
- What would you do differently if the teammate had *not* mentioned AI?

**Deliverable:** Add your reflection to your document.

#### Activity 2: Drafting Team AI Guidelines (12 minutes)

**Objective:** Think critically about how your team should approach AI usage and propose concrete guidelines.

**Task:**
You'll draft **team guidelines for responsible AI use** as if you're proposing them to an engineering team. This is practice for real conversations you'll have as an engineer (or potential team lead).

**Scenario:**
Your team is having its first conversation about AI tools. You've been asked to propose **initial guidelines**. You won't cover everything—just establish principles and a few concrete practices.

**Step 1: Brainstorm Team Context (2 minutes)**
Consider: **What kind of team are you imagining?**
- Startup (fast-paced, move fast and break things)?
- Enterprise (compliance-heavy, security-critical)?
- Regulated industry (financial services, healthcare)?
- Open-source project?
- Your current team if you're on one?

Write 1-2 sentences describing the team context.

**Step 2: Draft Core Principles (4 minutes)**
Propose 3-4 **core principles** for your team's AI use. These should be values-based, not rules.

Example format:

```plaintext
Principle 1: Transparency First
Description: We disclose AI usage in code review and pull requests. 
This builds trust and ensures code quality isn't sacrificed for speed.

Principle 2: Ownership & Accountability
Description: The developer remains responsible for all code, even if AI-assisted. 
We don't use "AI wrote it" as an excuse for poor quality or security issues.

Principle 3: Thoughtful Application
Description: AI is a tool for accelerating routine work, not a replacement for learning.
We use AI strategically—not everywhere, just where it adds value.

Principle 4: Continuous Learning
Description: We regularly revisit these guidelines as we learn what works (and what doesn't).
```

**Deliverable:** List your 3-4 principles with descriptions. Aim for 1-2 sentences per principle.

**Step 3: Identify Specific Practices (4 minutes)**
For each principle, propose 1-2 **concrete practices**. These are the actual things the team will do.

Example format:

```plaintext
Principle 1: Transparency First
Practice 1a: In PR descriptions, note "AI-assisted" for any code or documentation 
generated with significant AI help. Example: "Used ChatGPT to generate initial tests (reviewed and adjusted)."
Practice 1b: During code review, if AI-assisted code looks unusual, reviewers can ask: 
"Help me understand the approach here"—creator explains their reasoning.

Principle 2: Ownership & Accountability
Practice 2a: No AI-generated code ships without human verification (tests pass, logic verified).
Practice 2b: Security-critical code (auth, payment, data handling) should have explicit 
human review before merge—AI usage noted.
```

**Deliverable:** Draft 5-6 specific practices (2-3 per principle). Keep each to 1-2 sentences.

**Step 4: Identify Boundaries (2 minutes)**
Think about where AI use might be **problematic or inappropriate** for your team. Propose 1-2 explicit boundaries.

Example format:

```plaintext
Boundary 1: Interview Take-Homes & Assessments
Reason: We need to see each candidate's own thinking and problem-solving
Boundary 2: Initial architecture design & technical decisions
Reason: These decisions shape the codebase; we need deliberate human reasoning
```

**Deliverable:** Add 1-2 boundaries to your document.

**Step 5: Reflect on Your Guidelines (1 minute)**
Write a 2-3 sentence reflection:
- Would you feel comfortable proposing these guidelines to your team?
- What concern might teammates raise, and how would you address it?
- What would you change if this were a regulated industry (healthcare, finance)?

**Deliverable:** Add your reflection to your document.

### Guided Practice

**Time Allocation:** 45 minutes
Program staff will facilitate interactive discussion and live demonstrations. Come ready to participate, share perspectives, and learn from peers' experiences.

#### Part 1: Code Review Debrief & Communication Strategies (15 minutes)

**Discussion:**
Program staff will present a **framework for communicating about AI in code review**:

**The Three-Part Communication Model:**
1. **Disclose:** Be clear and specific about what AI helped with
  - Example: "I used ChatGPT to generate initial tests, then reviewed and adjusted each one"
  - Not: "AI wrote this" or (silently leaving AI usage unmentioned)
2. **Explain Your Thinking:** Show that *you* verified and own the output
  - Example: "I reviewed the logic, ran the tests, and verified it handles edge cases X, Y, Z"
  - Not: "I just used what AI gave me"
3. **Invite Collaboration:** Frame feedback as team learning, not criticism
  - Example: "I'd love feedback on this approach. Is there a better way?"
  - Not: "Is this good or bad?" (puts reviewer in defensive position)

**Participant Activity (12 minutes):**
Program staff will present **3-4 realistic code review scenarios** (varying in complexity and AI usage). You'll work in **small groups (3-4 people)** for 10 minutes to:
1. Read each scenario (e.g., "You submitted a UI component built with AI, but the reviewer questions its performance")
2. Draft a response using the three-part communication model
3. Discuss: "What's the tone? Does it build trust or sound defensive?"

**Example scenario provided:**

```plaintext
Scenario: PR Comment from Reviewer
"I see this component does a full re-render on every keystroke. 
That's going to be slow. Was this generated by AI?"

Your response should:
- Acknowledge the valid concern (they're right about performance)
- Explain your process (what did you do?)
- Propose a solution
- Invite collaboration (not defensiveness)
```

**Example good response:**
"Good catch! I used AI to generate the initial component structure, but I should've
optimized it before submitting. You're right—unnecessary re-renders will hurt performance.
I'll add useMemo and useCallback to prevent that. Thanks for the review!"

**Groups present findings** (program staff pick 1-2 groups to share their response).

**Wrap-Up (3 min):**
Highlight common themes. Clarify what transparency *doesn't* mean (you don't need to justify AI use or apologize). Address misconceptions about blame or responsibility.

**How to Show Up for Yourself, Prepared to Do the Work!:**
- Come with Activity 1 in mind—you've already drafted feedback
- Listen actively to how peers communicate in different scenarios
- Consider: "How would I respond in each scenario?" before groups present
- Ask clarifying questions if you're unsure how to handle a similar situation

#### Part 2: Team Dynamics & Governance Discussion (15 minutes)

**Discussion:**
Program staff will present **real-world perspectives on AI team dynamics**:

**Common Team Concerns About AI:**
1. **"Are people actually thinking, or just copying AI output?"**
  - Reality: This is a valid concern; the answer is clear code review and discussion
  - Mitigation: Encourage code review discussions about *why* this approach
2. **"Will AI replace senior engineers' roles if juniors use it?"**
  - Reality: AI accelerates routine work; what matters is learning and ownership
  - Mitigation: Frame AI as a tool for learning faster, not replacing expertise
3. **"Security/compliance issues—what if AI generates vulnerable code?"**
  - Reality: Valid concern for regulated industries; requires stricter review
  - Mitigation: Implement extra scrutiny for security-critical code
4. **"How do we ensure AI doesn't make us dependent?"**
  - Reality: Balance and intentional learning prevent this
  - Mitigation: Rotate who reviews code, require explanations, don't use AI for everything

**Participant Activity (12 minutes):**
You'll engage in **structured discussion** about team dynamics. Program staff will prompt:
1. **"Which concern resonates most with you, and why?"** (Chat or unmute to share)
2. **"What would you do if a teammate was uncomfortable with AI?"** (Small group discussion, then share)
3. **"How would you explain AI usage to someone skeptical?"** (Think-pair-share activity)

Program staff will collect responses and highlight diverse perspectives. The goal isn't consensus—it's understanding different viewpoints.

**Wrap-Up (3 min):**
Synthesize themes. Emphasize: **Team trust isn't about banning AI; it's about transparency and ownership.**

**How to Show Up for Yourself, Prepared to Do the Work!:**
- Come with Activity 2 guidelines in mind—you've thought about team context
- Be honest if you have concerns about AI—this is the space to voice them
- Listen to peers' perspectives without dismissing; different teams need different approaches
- Jot down one insight about team dynamics you hadn't considered before

#### Part 3: Guidelines Peer Review & Refinement (15 minutes)

**Small Group Activity:**
You'll pair with **one peer** to review each other's independent practice work.

**Exchange:** Share your Activity 1 and Activity 2 documents.

**For Activity 1 (Code Review) — 7 minutes per person:**
Peer review focuses on:
1. "Your feedback was effective because [specific observation about tone or clarity]"
2. "I think your concern about [specific issue] is important because..."
3. "The way you disclosed AI usage felt [transparent/defensive/collaborative] because..."
4. Suggestion: "One thing you might also flag: [idea] because..."

**For Activity 2 (Team Guidelines) — 8 minutes per person:**
Peer review focuses on:
1. "Your principle about [principle name] makes sense for [team context] because..."
2. "I think the practice of [specific practice] would work well because [reason]. Though I wonder about [edge case]..."
3. "Your boundary around [boundary name] is smart because..."
4. Reflection question: "If you proposed these guidelines to your actual team, how would they react? What pushback would you expect?"

**How to Show Up for Yourself, Prepared to Do the Work!:**
- Come with your completed independent practice document
- Listen actively to your peer's approach to communication and team governance
- Offer specific feedback using the structure above
- Share one insight you had from this peer review that you'll carry forward

### Showing Up for Yourself During Session

**Before the Session:**
- Complete both Activity 1 and Activity 2. Both are important for team collaboration perspective.
- Think about: *Have you had a real experience where a tool caused team conflict? How was it resolved?*
- Have your AI tool open if you'd like to practice live during the session.
- Consider your current or ideal team context as you draft guidelines in Activity 2.

**During the Session:**
- In Part 1 (Code Review Communication), pay close attention to tone. This is often where trust is built or lost.
- In Part 2 (Team Dynamics), listen for concerns different from yours. Everyone brings a valid perspective.
- In Part 3 (Peer Review), be curious about how your peer approached team guidelines—they might think about things you didn't.
- Take notes on one thing a peer said that changed your perspective about collaborative AI use.

**After the Session:**
- Revisit your team guidelines. Would you actually propose them? What would you refine based on today's discussion?
- If you're on a team, share what you learned today with teammates (in a non-prescriptive way). Gauge their perspective.
- Weeks 5 and 6 build on this foundation by focusing on **productivity & workflow** and **career long-term strategy**. Carry your collaborative mindset forward!

**Key Takeaway:**
AI doesn't eliminate the need for trust and communication in teams—it amplifies it. Engineers who are transparent, thoughtful, and collaborative about their AI usage are more trusted and more effective.
